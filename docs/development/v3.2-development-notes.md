# v3.2 — Per-SDK outgoing-offload routing

> **Status:** in flight. v3.1.0 ships first (Hub utility); v3.2.0 closes
> the SDK-side loop so any DVAI-bridge consumer app — not just the
> TypeScript core — can transparently offload inference to a paired
> peer (another phone running the SDK, or the Hub utility).

## Product flow (per user direction, 2026-05-08)

The intended runtime flow inside any SDK consumer app:

```
┌────────────────────────────────────────────────────────────────┐
│ 1.  init(context)                                              │
│ 2.  detectDeviceHints()      → DeviceCapabilityHints           │
│ 3.  heuristicTokPerSec(...)  → estimated tok/s (NO model load) │
│ 4.  optional: synthetic compute benchmark to refine            │
│ 5.  DECISION:                                                  │
│       a. tok/s < HARD_FLOOR (default 3)                        │
│            → throw HardwareTooWeak                             │
│            → SDK shows system popup ("device too weak")        │
│            → no model download, no server start                │
│       b. tok/s < OffloadConfig.minLocalCapability              │
│            → enter OFFLOAD_ONLY mode                           │
│            → skip model download/load                          │
│            → start proxy + discovery + pairing only            │
│       c. tok/s ≥ minLocalCapability                            │
│            → start normally (model + proxy + discovery)        │
│ 6.  Runtime per-request decision in the proxy:                 │
│       a. local enabled & capable → serve locally               │
│       b. peers paired & capable  → forward (HMAC-signed)       │
│       c. neither                  → 503 + Retry-After          │
└────────────────────────────────────────────────────────────────┘
```

**Key constraints:**

- **Capability test must NOT require a downloaded model.** Today the
  `probeCapability()` helper in `@dvai-bridge/core` calls
  `backend.chatCompletion()`, which presumes the model is loaded.
  v3.2 adds a pre-load assessment path that uses the heuristic alone,
  optionally refined by a synthetic compute benchmark (matrix-multiply
  timing or similar — not a real model).
- **Hard-floor abort is system-popup, not silent.** The user must see
  why the app stopped working; an OpenAI-shaped 503 wouldn't reach UI
  for a consumer app that wraps a chat client.
- **Offload-only mode skips model download entirely.** A device with
  4 tok/s shouldn't waste 2 GB of bandwidth on a model it'll never
  load locally.
- **Zero consumer-API change.** The SDK's `start()` returns the same
  `BoundServer { baseUrl, port }` shape. The `baseUrl` now points at
  the proxy — but the consumer never knows.

## Architecture — pre-routing proxy (option A)

Each native SDK currently exposes an HTTP server owned by the
backend (Llama / MediaPipe / LiteRT / etc.). v3.2 reshapes this:

```
Before (v3.1):
  consumer app → http://127.0.0.1:<backend_port>/v1/* (native server)

After (v3.2):
  consumer app → http://127.0.0.1:<proxy_port>/v1/*
                            │
                            ├─ if local: forward to <backend_port>
                            └─ if offload: HMAC-sign + forward to peer
```

The native backend still binds its own port, but on the loopback
interface only and unadvertised. The proxy is the only port the
consumer (or the rest of the LAN, if `discoverLAN: true`) sees.

### Why a proxy and not an OkHttp / URLProtocol interceptor?

Considered three options:

| Option | Consumer code change | SDK effort |
|---|---|---|
| A. Pre-routing proxy | **none** | high (proxy in 3 native langs) |
| B. OkHttp / URLProtocol interceptor | one-line `.addInterceptor()` | low (per-SDK, not too bad) |
| C. Hybrid (both A and B exposed) | none + opt-in advanced | medium |

User picked option A explicitly. The "developer deals with as
minimum code change as needed" requirement combined with the
desire for the SDK to handle hardware-too-weak / offload-only
decisions automatically only fits option A — interceptors require
the consumer to opt in by attaching them.

## Per-SDK breakdown

The work is split into seven milestones; each lands as its own commit
on `main` once locally validated. **No CI burns until M1+M7 are
locally green** (lessons from rc1–rc4).

### M1. Core (TS) foundations

Add to `packages/dvai-bridge-core/src/`:

- **`capability/precheck.ts`** — `assessCapability(opts)` runs
  `detectDeviceHintsAsync()` + `heuristicTokPerSec()` with no model
  loaded. Returns a `PrecheckResult { tokPerSec, mode: "ok" |
  "offload-only" | "too-weak" }`.
- **`offload/types.ts`** — extend `OffloadConfig` with:
  - `hardwareMinimum: number` (default 3 tok/s — below this, abort).
  - `onHardwareTooWeak?: (estimate) => void` — host hook for the
    system popup. Default behavior: SDK throws `HardwareTooWeak`.
- **`server/lifecycle.ts`** — new pre-start gate that runs
  `assessCapability()` before any backend init; routes the runtime
  into `ok` / `offload-only` / `too-weak` paths.

Estimate: 1–2 days. Scope-creep risk: low (mostly composition of
existing primitives).

### M2. Android — pre-routing proxy

Add to `packages/dvai-bridge-android/`:

- **Pre-routing proxy** in Kotlin (Ktor or NanoHTTPD — leaning Ktor
  for SSE-streaming support). Binds the public port; forwards local
  requests to the native backend's loopback port; forwards offload
  requests with HMAC-signed headers to paired peers.
- **Lifecycle reorder** in `DVAIBridge.start()`:
  1. Run capability precheck before `BackendSelector.resolve()`.
  2. If `too-weak` → `applicationContext` shows a system AlertDialog
     (using a foreground activity if present, falling back to a
     Notification with action), then throws `DVAIBridgeError.
     HardwareTooWeak`.
  3. If `offload-only` → skip the per-backend `start*()` calls;
     bring up only `initOffload()` + the proxy in offload-only mode.
  4. Otherwise → existing path.
- **Update `examples/android-llama`** to drop the manual OkHttp call
  to Hub and instead let the SDK's proxy handle the routing.

Estimate: ~1 week.

### M3. iOS — pre-routing proxy

Same shape as M2. `GCDWebServer` for the proxy (mature, SSE-friendly)
or `NWConnection` listener if we want to avoid the dependency.
System popup via `UIAlertController` from the active scene.

Estimate: ~1 week.

### M4. .NET — pre-routing proxy

Easiest of the three native SDKs because Kestrel is already in the
ASP.NET Core ecosystem. The proxy is essentially a YARP / minimal-API
forwarder. System popup uses `MessageBox` on Windows, `NSAlert` on
Mac Catalyst, `AlertDialog` on Android (.NET MAUI exposes
`Application.Current.MainPage.DisplayAlert`).

Estimate: 3–5 days.

### M5. React Native — TurboModule glue

The TurboModule already delegates to native iOS + Android. After
M2 + M3 land, the JS-side change is: surface
`OffloadConfig.hardwareMinimum`, surface a `pairingRequest` event
shape (already there), surface a new `hardwareTooWeak` event /
promise rejection.

Estimate: 2 days.

### M6. Flutter — Pigeon glue

Same shape as M5 but through the Pigeon channel.

Estimate: 2 days.

### M7. Public docs + examples

- `docs/guide/distributed-inference.md` — add "From the SDK side"
  section explaining the pre-routing proxy + capability precheck.
  Update the wire-protocol section if any HMAC / header conventions
  changed.
- `docs/guide/{android-native,ios-native,react-native,flutter,
  dotnet}-sdk.md` — each gets an "Outgoing offload" subsection with
  the platform's idiomatic config snippet (e.g.,
  `OffloadConfig(enabled: true, hardwareMinimum: 3, ...)`).
- Update `examples/android-llama` README to reflect that it now
  goes through the SDK rather than calling Hub directly.
- Bump version snippets to `3.2.0` in every install snippet.
- `CHANGELOG.md` — `[3.2.0]` section listing the new behavior.

Estimate: 2–3 days.

## Validation strategy (pre-CI)

- **M1**: TS unit tests against the new `assessCapability()` and
  `lifecycle.ts` paths. Mock the heuristic to drive each branch.
- **M2/M3/M4**: instrumented test on a real device per platform —
  no shortcut. Checks the three paths (ok, offload-only, too-weak)
  by overriding `OffloadConfig.hardwareMinimum` and
  `OffloadConfig.minLocalCapability` to force each branch.
- **M5/M6**: same end-to-end test against an emulator.
- **No CI runs** until all six milestones are locally green per
  the rc1–rc4 lesson. The v3.2.0-rc1 tag is the first CI iteration.

## Open questions

- Should the synthetic compute benchmark (matrix-multiply timing)
  ship in M1, or is the heuristic enough? Heuristic-only ships
  faster; benchmark refines accuracy on misclassified devices.
  Default: heuristic-only for v3.2.0; revisit after dogfood.
- For offload-only mode with no peers paired yet: does the SDK
  enter a "discovery loop, prompting the user to pair" mode, or
  does it return 503 immediately and rely on the host app to
  surface a UI? Default: 503 + a `peers_required` error code so
  the host can route a friendly message; no inline pairing UX in
  the SDK.
